# Phase 9 — UI Design Contract

> Visual and interaction contract for the popup pairing flow additive layer.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope of this spec:** Only the pairing state machine layered into the
> existing popup. All other popup capabilities (host list, Refresh, + Add host,
> Review Mode toggle, Show-hints checkbox, routing line, manual token field as
> fallback) are preserved exactly and are NOT redesigned here.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla DOM, no component library) |
| Preset | not applicable |
| Component library | none — vanilla DOM inside WXT popup HTML page |
| Icon library | none — text labels and Unicode glyphs only |
| Font | system-ui, -apple-system, sans-serif (inherited from body rule in popup.css) |

Source: popup.css `body { font-family: system-ui, -apple-system, sans-serif; }` — no change.

## Color

Existing popup color palette — matched exactly. No new palette colors.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #f8f8f8 | Body background, routing section, add-form strip |
| Secondary (30%) | #ffffff | Header, host list, empty state, toggle section, pairing banner |
| Accent (10%) | #111111 | Primary action button fill (Review Mode / Pair button) |
| Success | #22c55e | Paired connection dot; success status text |
| Destructive | #dc2626 | Exit Review Mode button; error/failed pairing text; remove host hover |
| Warning | #d97706 | Grant-mode state button (existing .sfx-grant-mode) |
| Muted text | #666666 | Host port label, host summary count, hint text |
| Border | #e0e0e0 | Section dividers |
| Border subtle | #f0f0f0 | Row dividers within host list |

Accent (#111111) reserved for:
- "Pair with host" button (pairing state: not-paired, host reachable)
- "Enter Review Mode" button (existing, unchanged)

Success color (#22c55e) reserved for:
- `sfx-dot--green` connection dot on a paired host row
- `.sfx-pairing-status--paired` status text "Paired"
- The paired status pill background (see State 5 below)

NOTE on Phase 8 UAT finding: The existing Apply button flashes `✓` text on `.sfx-token-btn:disabled` with `color: #16a34a` (green). This is ambiguous — users confused it with a send-success confirmation. The new "Paired" success state MUST NOT reuse this `✓` symbol alone. Use "Paired" as the word (not just a checkmark), paired with a persistent green dot. The Apply `✓` flash is still used for the token Apply button but is intentionally scope-limited to that one affordance and is ephemeral (disappears after 900ms).

## Pairing State Machine — Element Inventory

### New HTML element: `#sfx-pairing-banner`

A single `<section>` inserted immediately after `#sfx-header` and before `#sfx-add-form`.
It is the only structural addition to `index.html`.

```html
<section id="sfx-pairing-banner" hidden aria-label="Pairing status">
  <div id="sfx-pairing-status" role="status" aria-live="polite" aria-atomic="true">
    <!-- Dynamic content injected by main.ts pairing logic -->
  </div>
</section>
```

The `role="status"` + `aria-live="polite"` + `aria-atomic="true"` on `#sfx-pairing-status`
ensures screen readers announce state transitions without interrupting current speech.

`#sfx-pairing-banner` is `hidden` by default and shown only in states 1–4 below.
State 5 (already paired, quiet) does not show the banner — pairing is communicated
via the existing green dot on the host row.

---

### State 1: Not paired — host discoverable via native messaging

**Trigger:** Popup opens, `chrome.runtime.sendNativeMessage` has not been called yet
(or previous pairing cleared), and `sfxRegistry` is empty.

**Banner visibility:** Shown.

**Layout:**
```
┌─────────────────────────────────────────────────────┐
│ stickyfix            0 hosts   [Refresh]  [+]        │  ← existing header
├─────────────────────────────────────────────────────┤
│  Host found — click to pair                          │  ← #sfx-pairing-banner
│  [  Pair with host  ]                                │
└─────────────────────────────────────────────────────┘
```

**Element spec:**
- `#sfx-pairing-status` text: `"Host found — click to pair"`
  - Font: 12px, weight 400, color #444
- Button `#sfx-pair-btn`:
  - Text: `"Pair with host"`
  - Style: matches `#sfx-review-btn` exactly — `background: #111`, `color: #fff`,
    `font-size: 13px`, `font-weight: 600`, `border-radius: 5px`, `padding: 7px 14px`,
    `width: 100%`, `border: none`
  - Hover: `background: #333`
  - Focus: `outline: 2px solid #111; outline-offset: 2px` (meets WCAG 2.1 §2.4.7)
  - `aria-label`: `"Pair with host"`

**Note:** If `sfxRegistry` is non-empty on popup open (returning user, state 5), skip
this state entirely and go directly to state 5 — do not show the banner or fire
`sendNativeMessage` automatically. Auto-pairing on popup open is attempted ONLY when
the registry is empty (fresh install or cleared state).

---

### State 2: Pairing in progress

**Trigger:** User clicks "Pair with host" button (or auto-attempt fires on popup open
when registry is empty and native host is known to be registered).

**Banner visibility:** Shown.

**Layout:**
```
┌─────────────────────────────────────────────────────┐
│ stickyfix            0 hosts   [Refresh]  [+]        │
├─────────────────────────────────────────────────────┤
│  Pairing…                                            │
│  [ (spinner) Pairing… ]   ← button disabled          │
└─────────────────────────────────────────────────────┘
```

**Element spec:**
- `#sfx-pairing-status` text: `"Pairing…"`
  - Font: 12px, weight 400, color #666
- `#sfx-pair-btn`:
  - Text: `"Pairing…"` (matches existing Refresh → "Refreshing…" and Probe → "Probing…" patterns)
  - `disabled` attribute set
  - Style: `background: #999; cursor: not-allowed` (matches `#sfx-review-btn:disabled`)
- Spinner: CSS-only rotating `::before` pseudo-element on `#sfx-pair-btn.sfx-pairing`.
  - 10px diameter circle, `border: 2px solid #ccc; border-top-color: #fff`
  - Animation: `sfx-spin 0.6s linear infinite`
  - `@keyframes sfx-spin { to { transform: rotate(360deg); } }`
  - Displayed inline-block, `margin-right: 6px`, `vertical-align: middle`
  - Class applied: `#sfx-pair-btn.sfx-pairing`

---

### State 3: Paired successfully (first time, transition state)

**Trigger:** `sendNativeMessage` callback receives `{ type: 'TOKEN', token, port, name, notesDir }`,
SW persists to `sfxTokens` + `sfxRegistry`, then re-renders host list.

**Duration:** This state is transient — after 1200ms it transitions automatically to
state 5 (already paired, quiet). The banner is hidden and the host list updates to
show the new host row with a green dot.

**Banner visibility:** Shown for 1200ms, then hidden.

**Layout:**
```
┌─────────────────────────────────────────────────────┐
│ stickyfix            1 host    [Refresh]  [+]        │
├─────────────────────────────────────────────────────┤
│  ● Paired with <host name>                           │  ← #sfx-pairing-banner (transient)
├─────────────────────────────────────────────────────┤
│ ●  myproject       :39241   [token ••••] [Apply][Clear][−] │  ← normal host row
└─────────────────────────────────────────────────────┘
```

**Element spec:**
- `#sfx-pairing-status` text: `"Paired with <host name>"` — use the `name` field from
  the native messaging response, not a generic string.
  - Font: 13px, weight 600, color #16a34a (existing success green, matches `sfxTokens` Apply confirm)
  - Prefix: a static `●` glyph (U+25CF, filled circle), color #22c55e, `margin-right: 6px`
    — this is the same green dot used on host rows and is unambiguous as "connected"
  - DO NOT use `✓` (checkmark) as the sole indicator — per Phase 8 UAT finding,
    standalone `✓` is ambiguous. The word "Paired" + host name is required.
- `#sfx-pair-btn`: hidden (set `hidden` attribute) — the action is complete.
- After 1200ms: hide `#sfx-pairing-banner`, re-render host list (it is already populated).

---

### State 4: Pairing failed — native host not installed or error

**Trigger:** `chrome.runtime.lastError` is set, or the response is not `{ type: 'TOKEN' }`,
or the native host is not registered.

**Banner visibility:** Shown (persistent — no auto-dismiss).

**Layout:**
```
┌─────────────────────────────────────────────────────┐
│ stickyfix            0 hosts   [Refresh]  [+]        │
├─────────────────────────────────────────────────────┤
│  Auto-pair failed. Run:                              │
│  npx stickyfix init                                  │  ← inline code block
│  Then click Refresh, or enter your token below.      │
│  [  Retry  ]                                         │
├─────────────────────────────────────────────────────┤
│  (existing host list / empty state renders below)    │
└─────────────────────────────────────────────────────┘
```

**Element spec:**
- `#sfx-pairing-status`:
  - Line 1: `"Auto-pair failed. Run:"` — font 12px, weight 400, color #dc2626
  - Line 2: inline `<code>npx stickyfix init</code>` — font-family monospace (matching
    `#sfx-empty-hint code`), font-size 11px, background #f0f0f0, padding 1px 4px,
    border-radius 3px, color #333
  - Line 3: `"Then click Refresh, or enter your token below."` — font 11px, weight 400,
    color #666
- `#sfx-pair-btn`:
  - Text: `"Retry"` — same style as state 1 "Pair with host" button.
  - Clicking retries `sendNativeMessage` and transitions to state 2.
- The manual token field on each host row is the documented fallback. No additional
  disclosure text is needed in the banner beyond "enter your token below" — the
  existing token input is already visible in the host rows (or the user adds a host
  via the `+` form first).
- Error detail (chrome.runtime.lastError.message): displayed as an additional line
  in a `<details>` element below the Retry button.
  - `<summary>` text: `"Details"` — font 11px, color #888, cursor pointer
  - Body: raw error message, monospace 10px, color #888, word-break: break-all
  - Collapsed by default — technical users can expand; casual users ignore.

---

### State 5: Already paired — returning user

**Trigger:** Popup opens and `sfxRegistry` has at least one entry (populated from a
previous successful pairing or manual token entry).

**Banner visibility:** Hidden. The banner (`#sfx-pairing-banner`) remains `hidden`.

**Visual treatment:** The existing green dot (`sfx-dot--green`) on the host row
communicates paired status. No nag, no banner, no auto-re-pair attempt.

**Exception:** If the green dot is grey (host unreachable on the last Refresh cycle),
the user may click Refresh to re-probe. The pairing banner is NOT shown for this case —
host connectivity and token pairing are distinct concerns. A grey dot means the HTTP
host on that port is not responding; it does not mean the token is invalid.

## Interaction Contract

### Auto-pair on popup open (fresh install path)

1. `init()` reads `sfxRegistry`.
2. If registry is empty AND `chrome.runtime.getPlatformInfo` is available (i.e., extension context is valid):
   - Show `#sfx-pairing-banner` immediately.
   - Set status to state 1 ("Host found — click to pair").
   - Do NOT auto-fire `sendNativeMessage` on popup open. The button is the trigger.
   - Rationale: auto-firing native messaging on popup open creates a confusing
     experience if the native host is not installed — the user sees a flash of
     "Pairing…" then immediately "Auto-pair failed" with no action taken. Button-driven
     is more predictable and matches the "click to pair" success criterion (ROADMAP SC-2:
     "one-click 'Pair'").
3. If registry is non-empty: hide banner (state 5).

### Pair button click sequence

1. Button clicked → disable button, add `.sfx-pairing` class, set status text to "Pairing…" (state 2).
2. Call `chrome.runtime.sendNativeMessage('com.stickyfix.host', { type: 'GET_TOKEN' }, callback)`.
3. On success (`response.type === 'TOKEN'`):
   - Remove `.sfx-pairing` class.
   - Set status class to `.sfx-pairing-status--paired`, text to "Paired with <name>" (state 3).
   - Hide `#sfx-pair-btn`.
   - Re-render host list (now has 1 entry with green dot).
   - After 1200ms: set `#sfx-pairing-banner` `hidden`.
4. On failure (`chrome.runtime.lastError` set or response type mismatch):
   - Remove `.sfx-pairing` class.
   - Set status class to `.sfx-pairing-status--failed` (state 4).
   - Set line 1 text, render `<code>` block, set line 3 text.
   - Re-enable button with text "Retry".
   - If `chrome.runtime.lastError.message` is non-empty: populate `#sfx-pairing-error-detail` and show `<details>`.

### Focus management

- On popup open in state 1: `#sfx-pair-btn` receives focus automatically (single primary
  action on fresh install, equivalent to a form's default-focused submit button).
  Use `#sfx-pair-btn.focus()` at end of `init()` when banner is shown.
- On state 4 (failure): `#sfx-pair-btn` ("Retry") receives focus.
- On state 3 → state 5 transition: focus moves to `#sfx-review-btn` (the next primary
  action after pairing).

### Preserved behaviors (no regression)

| Feature | Preserved How |
|---------|---------------|
| Host list rendering | `renderHosts()` unchanged; called after successful pairing to show new host row |
| Refresh button | `doRefresh()` unchanged; always visible in header |
| + Add host by port | `addBtn`/`addForm` logic unchanged; always available |
| Manual token field per host | Token input, Apply, Clear buttons unchanged; visible on every host row |

| Review Mode toggle | Unchanged — state, click handler, permission request ordering |
| Show-hints checkbox | Unchanged |

| Empty state text | Updated one line (npx stickyfix init); layout/style unchanged |

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Screen reader status announcements | `role="status" aria-live="polite" aria-atomic="true"` on `#sfx-pairing-status` |
| Button accessible name | `aria-label="Pair with host"` on `#sfx-pair-btn` (redundant with visible label; included for AT robustness) |
| Spinner not announced as text | Spinner is a CSS `::before` pseudo-element with no text content; not in the DOM |
| Error detail disclosure | `<details>/<summary>` is natively keyboard-accessible |
| Focus state | `#sfx-pair-btn:focus-visible` outline 2px solid #111 (meets WCAG 2.1 §2.4.7) |
| Disabled state communicated | `disabled` attribute on button; AT will announce "dimmed" or "unavailable" |
| Color not sole error indicator | Error state uses red text AND descriptive text ("Auto-pair failed. Run:") — not color alone |
| Motion / animation | `sfx-spin` animation is on a 10px spinner only; respects user preference via `@media (prefers-reduced-motion: reduce) { #sfx-pair-btn.sfx-pairing::before { animation: none; } }` |

## HTML Structure Delta (index.html additions only)

Insert the following `<section>` after `#sfx-header` and before `#sfx-add-form`:

```html
<!-- Pairing banner (hidden by default; shown by main.ts in states 1-4) -->
<section id="sfx-pairing-banner" hidden aria-label="Pairing status">
  <div id="sfx-pairing-status" role="status" aria-live="polite" aria-atomic="true"></div>
  <button id="sfx-pair-btn" type="button" hidden aria-label="Pair with host"></button>
  <details id="sfx-pairing-details" hidden>
    <summary>Details</summary>
    <p id="sfx-pairing-error-detail"></p>
  </details>
</section>
```

Note: `#sfx-pair-btn` is initially `hidden` and shown only when state 1 or state 4 is
active. The status div always exists in the DOM (for aria-live reliability) but is empty
until a state activates.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host page → shadow-root UI | The page can style/script around the injected UI; createShadowRootUi + `:host{all:initial}` isolates it |
| host response → toast/card text | `resp.file` / `resp.error` strings flow into rendered DOM |
| content script → SW relay | The free-note Send crosses into the privileged SW (sole HTTP client) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering (XSS) | toast.ts / card.ts rendering resp.file & resp.error | mitigate | All host-derived strings go into `textContent` only — never `innerHTML` (INVARIANT C); enforced in card.ts/toast.ts; asserted by grep gate in acceptance criteria. |
| T-04-04 | Elevation of Privilege | content-script localhost access | mitigate | card.ts never fetches 127.0.0.1; the free-note Send relays exclusively through `SFX_SEND_ANNOTATION` (SW is sole HTTP client). card.ts must not import lib/capture.ts either (text-only — D-06). |
| T-04-05 | Information Disclosure | page-controlled textarea content | accept | Comment text is developer-authored on their own machine; host validates payload shape + caps body at 12 MB (413 over) — no new exposure beyond the existing SFX_SEND_ANNOTATION contract. |
| T-04-06 | Spoofing | SFX_SEND_ANNOTATION origin | mitigate | SW derives origin from `chrome.tabs.get(tabId).url`, never the message body (existing T-03-01 anti-spoof reused unchanged). |
</threat_model>

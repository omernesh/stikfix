<objective>
Implement the service worker — the architectural seam this whole phase proves. background.ts owns ALL localhost HTTP: it discovers hosts (parallel probe), reconciles the registry by name+origin on every wake, routes the active tab's origin to a host via `lib/routing.ts`, runs the page self-id probe for step-3 resolution, and performs the single `fetch` to `127.0.0.1/annotation` with the host's token. The content script never touches localhost — it only sends messages here.

Purpose: This is EXT-05, the SW-as-sole-HTTP-client boundary, plus the discovery/routing/persistence machinery (EXT-04/06/08/09/10). Once this lands, Plan 04's chip Send and Plan 03's popup toggle have a working SW to talk to, making the end-to-end relay proof (Success Criterion 3) possible.

Output: a complete `entrypoints/background.ts` message router with handlers for ENTER_REVIEW, EXIT_REVIEW, GET_ROUTE, SEND_ANNOTATION, and REFRESH_HOSTS, all storage-backed and chrome-lifecycle-safe.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| page/content-script → SW | Page JS and the injected content script are untrusted relative to the SW; only the SW holds host_permissions and tokens |
| SW → host (127.0.0.1) | Token-gated HTTP boundary; the SW is the sole client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | SEND_ANNOTATION origin | mitigate | Derive origin from chrome.tabs.get(tabId).url in the SW, NOT from the message body — a page cannot spoof which host it routes to (Task 2) |
| T-03-02 | Information Disclosure | per-host tokens | mitigate | Tokens live only in chrome.storage.local (extension-scoped); page JS cannot read them; the token is attached only inside the SW fetch, never exposed to the content script |
| T-03-03 | Tampering | malicious page sending SEND_ANNOTATION | accept | Page JS / other extensions cannot send chrome.runtime.sendMessage to our SW; worst case is a note to the user's own notes dir (host token still required) — low value, local-only |
| T-03-04 | Elevation of Privilege | LNA/CORS bypass via content-script fetch | mitigate | Architecture forbids content-script localhost fetch; the SW relay is the only path (grep-enforced: no localhost fetch outside background.ts) |
| T-03-05 | Denial of Service | unbounded discovery probes | mitigate | Fixed 21-port range, parallel allSettled, 800ms AbortController timeout (Plan 01 discovery) |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | Phase 3 installs NO new packages (RESEARCH Package Legitimacy Audit confirms); no install task exists — gate N/A |
</threat_model>

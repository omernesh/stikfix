# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: Element-Note Mode + Rich Context Capture.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Tech context:** Vanilla DOM inside a WXT `createShadowRootUi` shadow root.
> No framework. No component library. CSS is shadow-scoped (`styles.css` via
> `cssInjectionMode:'ui'`). px units only (rem inherits from host page — pitfall 5).
> All existing Phase 4 surfaces (chip, FAB, card, toast) are the visual baseline
> this phase extends.

**Accent reserved for:**
- Send button background (`#1d6ed8`)
- Textarea focus ring (`outline: 2px solid #1d6ed8`)
- Re-map label hover (`color: #1d6ed8`)

**Orange accent (`#ff6b00` family) reserved for:**
- Hover-highlight overlay outline + fill (live DOM; hidden before capture)
- Canvas-drawn box on `+1.png` (post-capture; not live DOM)
- Picker button active state border + background indicator

**Rationale for orange:** orange (`#ff6b00`) is the conventional DevTools
inspection highlight color. It reads on white, dark, and highly saturated
page backgrounds and does not conflict with the blue accent used for
interactive actions. Contrast ratio of orange outline on white = ~3:1 (WCAG
AA large text; acceptable for a temporary, cursor-following overlay that
disappears on click, not a persistent interactive control).

Source: CONTEXT.md D-02 "DevTools-style"; phase context "high contrast on
light AND dark pages" mandate.

## Component Inventory — Phase 5 New Surfaces

### 1. Picker Button (`#sfx-picker-btn`) — ELEM-01

**Placement:** Appended inside `#sfx-chip` (the top-right connection chip),
after the existing route label and before the exit button. Existing chip
DOM is extended; chip layout already uses `display:flex; gap:8px`.

**Resting state:**
```
width: 28px; height: 28px          — touch target padded to 44px via padding or wrapper
border-radius: 50%                  — round pill matching FAB aesthetic
background: #f0f0f0
border: 1px solid #cccccc
cursor: pointer
font-size: 16px                     — emoji glyph 🎯
display: flex; align-items: center; justify-content: center
pointer-events: auto
```

**Active state (pick mode ON):**
```
background: #fff3e0                — sfx-picker-btn-active-bg
border-color: #ff6b00              — sfx-picker-btn-active-border
box-shadow: 0 0 0 2px rgba(255,107,0,0.25)  — glow ring
```

**Hover (resting):** `background: #e0e0e0` (matches `.sfx-chip-btn:hover`)

**Hover (active):** same active state (no change — button is already active)

**Disabled:** not applicable (button always clickable; clicking in active mode cancels)

**Aria:** `aria-label="Pick element"` resting; `aria-label="Cancel element pick (Esc)"` active; `aria-pressed="false"/"true"` to encode toggle state.

**Class:** `.sfx-chip-btn` base class + `.sfx-picker-btn` modifier. The base
class gives correct font, sizing, and border — only the modifier overrides
shape and active states.

### 2. Hover-Highlight Overlay (`.sfx-hover-highlight`) — ELEM-01

**Nature:** A single `<div>` appended to the shadow-root container. Absolutely
positioned at the hovered element's page-absolute `getBoundingClientRect()`
coordinates. Updated on every `mousemove` (throttled to `requestAnimationFrame`).
`pointer-events: none` — the underlying page element still receives the click.

**Visibility toggle:** `display: none` (hidden) / `display: block` (pick mode active + cursor over page element). Hidden immediately (synchronously) before `waitTwoRafs` + capture, then restored after.

**Positioning:**
```
position: fixed                   — follows viewport coords from getBoundingClientRect
left: {rect.left}px
top: {rect.top}px
width: {rect.width}px
height: {rect.height}px
box-sizing: border-box
pointer-events: none
z-index: 2147483646               — one below :host z-index; inside shadow root so effective z matches :host
```

Note: because this `<div>` lives inside the shadow root that already occupies
`z-index: 2147483647` on the page, an inner `z-index` of 1 suffices for
stacking within the shadow root. Use `z-index: 1` (not 2147483647) inside
the shadow root to avoid confusion. The `:host` already pins the whole
overlay above the page.

**Outline + fill:**
```
border: 2px solid rgba(255, 140, 0, 0.85)
background: rgba(255, 140, 0, 0.08)
border-radius: 2px                — barely rounded; matches DevTools
```

**Label (`.sfx-hover-label`):** Absolutely positioned below the overlay box.
If the element is within 24px of the bottom viewport edge, label flips above.

```
position: absolute
bottom: calc(100% + 4px)          — above box (default); flip to top: calc(100% + 4px) when near bottom
left: 0
background: rgba(0, 0, 0, 0.75)
color: #ffffff
font-size: 13px
font-weight: 400
line-height: 1
padding: 4px 8px
border-radius: 3px
white-space: nowrap
pointer-events: none
max-width: 240px
overflow: hidden
text-overflow: ellipsis
```

**Label content format:** `tag · WxH` e.g. `button · 120×40`

If element has an `id`, use `tag#id · WxH`. If element has a single
meaningful class (non-hashed, non-Tailwind utility), use `tag.class · WxH`.
Limit the prefix to ≤ 20 chars before ` · WxH` to avoid overflow. The
`WxH` uses `Math.round()` of `rect.width × rect.height` in CSS px (not DPR-scaled).

**Esc key:** Pressing Esc while pick mode is active exits pick mode, hides
the overlay, and resets the picker button to resting state. No card opens.

### 3. Element Post-It Card — Context Header (`.sfx-ctx-header`) — D-03 / ELEM-07

The element-mode card extends the existing `#sfx-card` structure. The card
itself keeps the same ID and class conventions; the element variant adds
`.sfx-card-element` modifier class on the root `#sfx-card` element.

**Focal-point intent:** The context header strip draws the eye first — it is
pre-filled with content and carries a distinctive warm off-white background
(`#f8f4f0`) that visually separates it from the white card body. This confirms
"the picker grabbed the right element" at a glance, then directs attention
downward to the (empty) textarea where the developer types, and finally to the
Send CTA. The reading order is: context confirmation → write comment → Send.

**Context header position:** Inserted between `.sfx-card-header` (drag handle) and `.sfx-card-body` (textarea). It is a read-only, non-interactive display strip.

**DOM structure:**
```html
<div class="sfx-ctx-header" role="note" aria-label="Element context">
  <span class="sfx-ctx-header-text">button.primary · "Save" · <LoginForm> · 120×40</span>
</div>
```

`textContent` only — XSS-safe (page-controlled strings from element inspection).

**Visual spec:**
```
padding: 4px 12px                  — matches card-body left padding for alignment
min-height: 28px
background: #f8f4f0               — sfx-ctx-header-bg
border-bottom: 1px solid #e8e0d8  — sfx-ctx-header-border
font-size: 13px
font-weight: 400
color: #444444                    — muted foreground
line-height: 1.4
overflow: hidden
white-space: nowrap
text-overflow: ellipsis
```

**Context summary format:** `shortSelector · "text" · <Component> · WxH`

- `shortSelector`: `tag`, `tag#id`, or `tag.class` — first meaningful token only, ≤ 20 chars
- `"text"`: first 40 chars of collapsed inner text in double quotes; omitted if empty
- `<Component>`: React component display name in angle brackets; omitted if undetectable
- `WxH`: rounded integer dims from `getBoundingClientRect()`
- Parts separated by ` · ` (space + middle dot + space)
- Entire string truncated to a single line via CSS `text-overflow: ellipsis`

**Card header label:** Changed from `"Free note"` to `"Element note"` for element-mode cards.

### 4. Canvas Highlight Box — `+1.png` artifact (D-02 / ELEM-08)

**Not a live DOM element.** Drawn onto the captured canvas after `captureTab`
returns. Specified here to lock the visual so the developer and executor
agree on the look of the screenshot artifact.

**Box spec (drawn at DPR-corrected coordinates):**
```
strokeStyle: '#ff6b00'            — opaque orange (sfx-canvas-box-stroke)
lineWidth: 2 * devicePixelRatio   — hairline at 1×; 4px at 2×; sharp at any DPR
fillStyle: 'rgba(255, 107, 0, 0.15)'  — translucent interior tint
```

**Drawing order:** fill first (transparent interior), then stroke (opaque border).
Coordinates: `rect.x * dpr`, `rect.y * dpr`, `rect.width * dpr`, `rect.height * dpr`
where `rect` is the page-absolute `getBoundingClientRect()` of the element
captured at click time (not at capture time — same rect, frozen on click).

**Label on canvas:** None. The `+1.png` box is unlabelled (label lives in the
`.md` frontmatter; no text overlay on the PNG). Keeps the screenshot clean
and avoids font-rendering complexity on canvas.

**Own-UI absence:** The live hover-highlight overlay div (`sfx-hover-highlight`),
the picker button active state, the post-it card, and the chip are all hidden
via `waitTwoRafs` before `captureTab` fires. The canvas box is then drawn
onto the returned canvas after capture — so the PNG shows only page content
plus the drawn box.

## Interaction Contracts

### Pick Mode State Machine

```
IDLE → [🎯 click] → PICK_ACTIVE
PICK_ACTIVE → [element click] → IDLE (card opens)
PICK_ACTIVE → [Esc key] → IDLE (no card)
PICK_ACTIVE → [🎯 click again] → IDLE (toggle cancel; no card)
PICK_ACTIVE + card open → not possible (single-shot D-01: pick exits on click)
```

**Cursor:** `cursor: crosshair` applied to `:host` during pick mode (overrides
`:host { cursor: default }` from baseline). Reverted on exit.

**Event isolation:** The content script adds `mousemove` + `click` listeners
on `document` (page scope) during pick mode. On exit (Esc, click, or toggle),
both listeners are removed. The click listener must call `e.stopPropagation()`
only enough to prevent the click from activating the page's own handler
(i.e. `e.preventDefault()` optional; `e.stopPropagation()` optional — depends
on page behavior; the element's `getBoundingClientRect()` and selector are
captured BEFORE any prevention). **Recommended:** capture context from `e.target`,
then do nothing to propagation — the page can handle its own click normally;
the extension opens the card independently.

### Keyboard Contract

| Key | Context | Action |
|-----|---------|--------|
| Esc | Pick mode active, no card | Exit pick mode, hide overlay, reset picker button |
| Esc | Card open (element or free) | Close card (existing Phase 4 behavior) |
| Ctrl+Enter / Cmd+Enter | Card textarea focused, Send enabled | Send note |
| Tab | Card open | Standard focus cycling within card (textarea → Send → Discard note) |

### Focus Management

- On pick mode entry: picker button `aria-pressed="true"` but focus stays
  wherever it was — the developer is now using their mouse to pick.
- On element click → card opens: focus moves to textarea (same as free-note
  card — `Promise.resolve().then(() => textarea.focus())`).
- On Esc (pick mode): picker button receives focus (`pickerBtn.focus()`) so
  keyboard users can confirm the mode change without hunting.
- On Esc (card): card removes; picker button receives focus if pick mode was
  the source; FAB receives focus if free-note was the source.

### Send Interaction (Element Mode)

Send button disabled rule for element-mode card: disabled only when textarea
is EMPTY after trim — same as free-note card. The context header is pre-filled
read-only; it does not count toward Send enablement. The developer writes their
comment into the textarea; the element context lands in the `.md` regardless.

Send button label: `"Send"` (same as free-note). In-flight: `"Sending…"`.
The secondary button is labelled `"Discard note"` — it removes the unsent card
(same card-state discard behavior as the free-note card; only the visible label
text differs from Phase 4).

**In-flight state during capture:**
1. Send button → disabled + `"Sending…"`. Discard-note button disabled. Textarea readOnly.
2. Hide all sfx UI from DOM visibility (`display:none` on card, chip, FAB,
   hover overlay if visible).
3. `waitTwoRafs()` — ensures repaint before capture.
4. `captureTab(tabId)` — full viewport dataURL via SW.
5. Draw highlight box on canvas.
6. Restore UI visibility.
7. Assemble payload, `SFX_SEND_ANNOTATION` relay.
8. On response: toast + close card, OR re-enable controls + error toast.

- `<Component>` omitted if React fiber walk is undetectable

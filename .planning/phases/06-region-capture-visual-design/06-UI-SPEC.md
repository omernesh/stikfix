# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Region Capture + Visual Design + Persistent Pins.
> Generated by gsd-ui-researcher.
>
> **Tech context:** Vanilla DOM inside a WXT `createShadowRootUi` shadow root (`<sfx-review-ui>`).
> No framework. No component library. CSS is shadow-scoped (`styles.css` via `cssInjectionMode:'ui'`).
> px units only — rem inherits from host page font-size (pitfall 5 / WXT FAQ).
> All Phase 4–5 surfaces (chip, FAB, card, toast, picker, hover-highlight) are the baseline.
> This contract EXTENDS Phase 5 (approved 2026-06-02). Nothing in Phase 5 is changed.
>
> **Phase 6 adds three new surfaces:** (1) region-capture scrim + marquee + camera button + thumbnail
> strip, (2) paper-aesthetic visual polish on existing card + header + toast surfaces, (3) persistent
> on-page pin markers with mode color, unread/read dot, hover preview, and view/edit/delete card.

existing toast `opacity 200ms ease` — keep motion minimal.

**Placement:** Appended inside `.sfx-card-header` (the drag-handle strip on every
card), after the card-header-label and before the right edge. Both free-note and
element-note cards get the camera button.

**Resting state:**

```
border-radius: 50%
background: #f0f0f0
border: 1px solid #cccccc
cursor: pointer

padding: 0

pointer-events: auto
margin-left: auto                  — push to right edge of header flex row
```

**Active state (marquee drag in progress):** Not applicable — the scrim and crosshair

**Hover (resting):** `background: #e0e0e0` — matches `.sfx-chip-btn:hover`.

it lives in the card header, not the chip).

### 2. Capture Scrim + Marquee Rect — CAM-02, CAM-03

**Scrim (`.sfx-cam-scrim`):**

The scrim is a full-viewport `<div>` appended to the shadow-root container.
It dims the page and provides the draggable surface for the marquee.

```
position: fixed
inset: 0
background: rgba(0, 0, 0, 0.35)    — 35% black overlay (CAM-02: "dims the page")
pointer-events: auto               — captures all pointer events during camera mode
cursor: crosshair                  — CAM-02 requirement
z-index: 1                         — within shadow root, below chip (see Z-order section)
```

The scrim is created on camera-button click, removed on drag-end, Esc, or cancel.
During the scrim's lifetime the card that triggered it remains in the DOM but is
visually behind the scrim — the developer cannot accidentally interact with it.

**Marquee rect (`.sfx-cam-rect`):**

A child `<div>` of the scrim that is resized in realtime during drag.

```
position: absolute                 — absolute within the fixed scrim (inset:0)
border: 2px solid #1d6ed8          — accent blue stroke (CAM-03 live feedback)
background: rgba(29, 110, 216, 0.12)  — translucent blue tint
display: none                      — hidden until dragstart
box-sizing: border-box
pointer-events: none               — scrim handles pointer events; rect is visual-only
```

**Marquee draw contract:**
- `dragstart`: show `.sfx-cam-rect`, set `left/top` to `event.clientX/Y`, `width/height = 0`.
- `dragmove`: update `left/top` to `min(startX, currentX)`, `width/height = |delta|`.
- `dragend`: if `width < 6 && height < 6` → cancel (CAM-03 threshold); else → capture.
- `keydown Esc` at any point → cancel; removes scrim, calls `onCancel`.

**Coordinate system:** `event.clientX/Y` from interact.js are viewport-CSS px.
These feed directly into `cropToRect(dataUrl, rect, window.devicePixelRatio)`.
No unit conversion needed — matches the existing `captureElementContext.rect` contract.

### 6. Pin View/Edit/Delete Card — PIN-05, PIN-06

Clicking a pin opens a card using the existing card surface (`#sfx-card`).
The pin card reuses the same card DOM structure with additions.

**Card mode:** `.sfx-card-pin` modifier on `#sfx-card`. Initial state is VIEW mode.

**Card header:** Uses mode color same as the source note (free = amber, element = blue).
Header label: `"Free note"` or `"Element note"` (same as compose cards).

**VIEW mode layout (`.sfx-card-pin-view`):**
```
.sfx-card-body contains:
  <div class="sfx-pin-body-text">      — note body text (textContent only)
    font-size: 14px; line-height: 1.5
    max-height: 200px; overflow-y: auto
  </div>
  <div class="sfx-thumb-strip">        — thumbnails (read-only; no × button in view mode)
    .sfx-thumb-img only (no .sfx-thumb-del)
  </div>

.sfx-card-footer (VIEW mode):
  [Edit]  [Delete]  [Close]
  Edit:   .sfx-cam-btn style — 12px, border, bg #f0f0f0
  Delete: .sfx-btn-delete — 12px, border: 1px solid #dc2626, color #dc2626, bg white
  Close:  same as existing cancel button style
```

**EDIT mode (`.sfx-card-pin-edit`):** Replaces view content in-place (no card close/reopen).
```
.sfx-card-body contains:
  [same context header strip if element note — read-only]
  <textarea> pre-filled with existing note body text — same as compose card textarea
  [thumbnail strip — thumbnails shown read-only; camera button added for new captures]

.sfx-card-footer (EDIT mode):
  [Save]  [Cancel]
  Save:   same style as Send button (#1d6ed8 bg, white text, 600 weight — label "Save")
  Cancel: same style as discard button
```

**In-flight EDIT state:** Save button → disabled + `"Saving…"`. Same pattern as Send.

**DELETE confirmation (PIN-06 — D-05 hard delete behind confirm):**

No modal dialog. Inline confirmation replaces the footer:
```
.sfx-card-footer (DELETE confirm):
  <span class="sfx-del-confirm-text">Delete this note and its screenshots?</span>
  [Confirm delete]  [Keep]
  Confirm: .sfx-btn-delete-confirm — bg #dc2626, color #ffffff, 12px, 600 weight
  Keep:    same as cancel/discard button
```

This avoids a blocking `window.confirm()` (which would be a cross-context call
from the shadow root and is not appropriate for an overlay tool).

**On successful delete:** pin element removed from shadow root; close card; show
success toast: `"Note deleted"`.

**On successful edit/save:** pin dot → unread (server re-marks unread per D-04);
close card; re-render pin with updated text in hover preview; show success toast:
`"Note saved"`.

## Interaction Contracts

### Camera Tool State Machine — CAM-01..04

```
CARD_OPEN (compose or view/edit)
  → [📷 click]             → CAM_SCRIM_ACTIVE
CAM_SCRIM_ACTIVE
  → [Esc]                  → CARD_OPEN (cancel; no capture)
  → [sub-6px drag + release] → CARD_OPEN (cancel; no capture)
  → [drag ≥ 6px + release] → CAPTURING
CAPTURING (async: hide UI → waitTwoRafs → captureTab → cropToRect → show UI)
  → [crop complete]        → CARD_OPEN (thumbnail appended to strip)
  → [crop error]           → CARD_OPEN (error toast: "Screenshot capture failed")
```

The card's own drag handle is NOT active while CAM_SCRIM_ACTIVE (scrim covers it).
Only one scrim can be active at a time (camera button disabled while scrim is shown).

**Camera button disabled state:** `opacity: 0.5; cursor: not-allowed` while a
capture is in progress. Re-enabled after crop completes or errors.

### Pin Lifecycle State Machine — PIN-01..06

```
REVIEW_MODE_ENTER
  → SW: SFX_LIST_ANNOTATIONS (once)
  → render all pins for URL path
CARD_SEND_SUCCESS (existing note)
  → re-fetch pins (SFX_LIST_ANNOTATIONS)
  → teardown + re-render all pins (reflects new note as pin)
PIN_CLICK
  → open pin view card
PIN_EDIT_CLICK (in view card)
  → switch to edit mode (in-place; no card close)
PIN_SAVE_CLICK (in edit mode)
  → SW: SFX_EDIT_ANNOTATION → PUT /annotation/<serial>
  → success: pin dot → unread; update hover preview text; toast "Note saved"
  → error: re-enable Save; toast error message
PIN_DELETE_CLICK (in view card)
  → show inline confirm footer
PIN_DELETE_CONFIRM_CLICK
  → SW: SFX_DELETE_ANNOTATION → DELETE /annotation/<serial>
  → success: remove pin; close card; toast "Note deleted"
  → error: restore footer; toast error message
REVIEW_MODE_EXIT
  → teardown all pins; remove scroll/resize listeners
```

### Z-order Contract (within shadow root)

The shadow root host occupies `z-index: 2147483647` on the page. Within the shadow root,
all children use internal z-index stacking. Declare these explicitly in `styles.css`:

| Element | z-index (internal) | Rationale |
|---------|-------------------|-----------|
| `.sfx-hover-highlight` | 1 | Below everything else; decorative |
| `.sfx-cam-scrim` | 1 | Same level as hover-highlight; covers page but not chip |
| `.sfx-pin` | 2 | Above scrim (pins must be clickable in normal mode, not during camera) |
| `.sfx-pin-preview` | 3 | Above pin itself |
| `#sfx-card` | 4 | Above pins |
| `#sfx-fab` | 5 | Above card |
| `#sfx-chip` | 6 | Always on top |
| `.sfx-toast` | 7 | Above chip |

**During camera mode:** The scrim at z:1 is below pins (z:2). This is intentional —
during camera mode, pins are not interactive (the scrim captures all pointer events).
Once the scrim is removed, pins become interactive again.

Note: these z-index values are all within the shadow root's isolated stacking context.
They do not compete with the host page's z-index values.

### Scroll/Resize Repositioning (PIN-02)

Element pins reposition on `scroll` and `resize` events. Throttle: 100ms
(approximately 10fps repositioning — sufficient for a dev tool; imperceptible lag
is acceptable). The throttle is a simple timestamp-delta guard (no lodash):

```typescript
let last = 0;
const onScroll = () => {
  const now = Date.now();
  if (now - last < 100) return;
  last = now;
  repositionAllElementPins();
};
window.addEventListener('scroll', onScroll, { passive: true });
window.addEventListener('resize', onScroll);
```

Both listeners are removed in `teardown()` which is called from `index.ts` `onRemove`.

## Shadow-DOM Isolation Contract (UI-01)

This is a hard constraint restated for Phase 6 clarity:

- **px units only** — zero `rem`, `em`, or `%` font-size anywhere in `styles.css`.
- **`:host { all: initial; }`** — preserved; Phase 6 adds no new `:host` properties.
- **No `innerHTML`** — all page-derived text (note body, pin hover preview, context header)
  set via `textContent` only.
- **No CSS variables from host page** — do not use `var(--*)` from outside the shadow root.
  All tokens are hardcoded hex/rgba values in `styles.css`.
- **Verification method:** Load the extension on `tailwindcss.com` (Tailwind-heavy) and
  `meyerweb.com/eric/tools/css/reset/` (CSS-reset-heavy). Pin and card rendering must
  be visually identical on both pages — no inherited font-size bleed, no inherited color,
  no reset interference.
